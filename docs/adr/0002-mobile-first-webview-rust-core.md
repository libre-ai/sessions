# ADR-0002 — Mobile-first WebView/PWA with a Rust core

- Status: Accepted
- Date: 2026-06-29
- Related: docs/adr/0001-product-architecture-and-boundaries.md, docs/specs/2026-06-28-frontend-dioxus-design-system-design.md, docs/status/2026-06-29-session-handoff.md

## Context

Presto-Matic's next product step after the `v0.1.0` backend/RAG/live-session baseline is a testable mobile-first surface. The product must stay sovereign and self-hostable, avoid app-store friction in the first iteration, and preserve the architectural invariant that product logic lives in Rust rather than being reimplemented in JavaScript/Swift/Kotlin.

cos-matic's portability doctrine is the reference shape: **one Rust core, bound or compiled to each target, never reimplemented**. For Presto-Matic this means the deterministic product contracts, protocol state, API DTOs, and client-side state machines stay in Rust crates. UI can be target-specific, but it consumes the Rust core.

## Decision

Build the first mobile testable product surface as a **mobile-first web/PWA served by `presto-server`**, implemented with a Rust/WASM client path and Rust crates shared with the backend protocol layer.

This ADR is amended by the Portal client-platform decision:

1. **MVP target:** browser/PWA mobile surface, testable with Playwright mobile profiles and real phones.
2. **Fast default:** Dioxus/PWA is the default wedge path when it keeps delivery simple and verifiable.
3. **Native first-class path:** SwiftUI/Compose clients are valid once a product need and local verification justify them; they consume Portal core/tokens rather than reimplementing product logic.

## Rust-core contract

The client architecture is:

```text
presto-core        shared protocol, public DTOs, serializable state
presto-client-core pure client state machines, API contracts, validation  [optional/new]
rumble-lm-ui       product UI primitives (renamed from presto-ui)
Portal             shared tokens, accessibility, i18n UI, web/native adapters
rumble-lm app      owner notebook PWA/WebView/native surface
rumble-lm join     guest live-join PWA/WebView/native surface
presto-server      HTTP/WS API, authz, RAG orchestration, static app host
```

Rules:

- Rumble LM client surfaces and `rumble-lm-ui` must not depend on `presto-server` as code.
- Browser JavaScript is glue generated by the Rust/WASM client, not a business-logic implementation.
- Shared client-platform semantics belong to Portal; product-specific UI stays in `rumble-lm-ui`.
- Any client state transition that affects correctness, authz display, live-session protocol, or RAG result interpretation belongs in Rust (`presto-core` or `presto-client-core`).
- Server remains authoritative for rights, timing, scoring, clearance, and grounding decisions.
- The web token is never readable by WASM/JS: `HttpOnly; Secure; SameSite=Strict` cookie + server-side `Sec-Fetch-Site` checks.

## First vertical slice

The first product vertical is the **authenticated personal notebook**, not the guest live client:

> Login via OIDC → personal space → ask a RAG question → render a grounded answer with cited source cards → logout/refresh.

Rationale: the notebook is the daily product surface and validates auth, mobile UI, RAG UX, and trust transparency before re-skinning the already-proven live engine.

The guest live client follows as the second vertical:

> Open join link → enter display name → answer a live question → reveal + leaderboard.

## Iteration gates

The mobile/WebView work must add gates as the surface appears:

- Rust portability: `cargo check` for client crates on `wasm32-unknown-unknown`.
- Quality: `cargo fmt`, `cargo clippy -D warnings`, workspace tests.
- Supply chain: existing `cargo-deny` + `cargo-audit` gates.
- Browser/mobile: Playwright mobile smoke tests once the PWA exists.
- Security: no token in JS/localStorage/sessionStorage/logs; CSRF same-origin checks; CSP before public testing.
- Sovereignty: no US hyperscaler, no proprietary frontend SaaS, no remote fonts/CDNs by default.

## Harness feedback loop

Presto-Matic should dogfood cos-matic's principle: when a repeatable automation/gate belongs to the harness rather than this product, create a GitHub issue in `constantin-jais/cos-matic` instead of embedding a local workaround.

A harness issue is required when we identify a generic missing capability such as:

- reusable `wasm32` portability gates for Rust-core products;
- milestone/issue scaffolding from a plan manifest;
- mobile/PWA/WebView agent domains;
- hard-gate vs observability goal reporting for product milestones;
- cross-repo issue linkage/reporting between a product repo and the harness repo.

Local Presto work may continue manually, but the missing generic harness capability must be tracked and linked from the relevant product issue.

Initial upstream feedback issues created while launching this milestone:

- cos-matic [#41](https://github.com/constantin-jais/cos-matic/issues/41) — mobile/WebView Rust-core product domain.

## Consequences

Positive:

- fastest path to a mobile-testable product without app-store friction;
- preserves one Rust source of truth for product logic;
- keeps self-hosting simple: the server can serve the PWA assets;
- leaves native/WebView shells open after auth and UX are proven.

Costs:

- Dioxus/WASM maturity and mobile browser quirks are now owned explicitly;
- accessibility and mobile design-system work must be built in-house;
- OAuth-in-native-WebView remains deferred, not solved by this ADR.

## Acceptance for the mobile MVP

The first mobile alpha is complete when:

- a user can test on a phone-sized viewport or real phone;
- login/logout works against a dev Keycloak;
- a RAG query returns a grounded answer with at least one citation;
- Playwright mobile smoke passes in CI;
- token secrecy and no-PII logging are verified;
- all required harness gaps discovered during the work have linked GitHub issues.
